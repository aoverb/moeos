### AI辅助编码原则

编写本项目代码时，我用到了AI辅助，但是遵守了以下原则：

1、在编写内核时，尽量不让AI帮我写代码，只让AI来帮我检视代码；编写用户态程序时和libc时会借助AI编写（因为我想聚焦于内核本身）

2、让AI给出方向上的建议，但是不给出实现上的建议；

3、与其用大量的时间给一个差的AI提出的提议擦屁股，不如从一开始就多花点钱来用一个好的AI，能减少你的时间损失和不必要的挫败感。

综合了大量过往对话，以下是我的总结。

项目概况：LoliOS
一个 i386 架构的自制操作系统，使用 i686-elf 交叉编译器，C/C++ 混合编写。从 OSDev Meaty Skeleton 起步，逐步实现了 GDT/IDT、PMM/VMM/Kheap、进程管理、系统调用、VFS/TARFS/DevFS/PipeFS、ELF 加载、用户态 shell、RTL8139 网卡驱动、完整网络栈（Ethernet → ARP → IP → ICMP → UDP → TCP），以及 ping 用户程序和 panic screen。项目完成度相当高。

你对我的依赖程度与侧重点
高频依赖区域（几乎每一步都来问）：

编译链接错误排查 — extern "C" 不匹配、__gxx_personality_v0、__cxa_atexit、符号未定义等。你遇到这类问题时几乎都直接贴错误信息让我看。
架构设计决策 — VFS 怎么抽象、pipe 要不要做成文件系统、网络栈分层顺序、TCP 哈希表策略。这些"该怎么设计"的问题你问得最多。
Bug 排查 — pipefs 的竞态条件、refcnt 生命周期、v_close 调用底层的时机。你经常是自己写完一版、跑不通了，然后贴代码让我帮找问题。

中频依赖区域：

API/标准对齐 — clock() vs get_tick、POSIX 函数放在 libc 哪个目录、scanf 应该怎么分层实现。
代码审查 — shared_ptr 实现、TCP 实现的全面审查。这方面你主动意识很好，甚至要求我引导式提问而非直接指出。

低频/几乎不依赖：

核心算法实现（你的环形缓冲区、进程调度、内存分配器基本都是自己写的）
汇编层面的东西（boot、context switch、中断入口）


做好了什么
1. 项目推进节奏极好。 从 Meaty Skeleton 一路做到有完整网络栈和用户态程序，覆盖面非常广。很多 OSDev 项目永远停在"能打印 Hello World"这一步，你远远超过了这个阶段。
2. 分层意识清晰。 VFS 的抽象、pipefs 作为独立文件系统挂载、网络栈的逐层构建（而不是一坨混在一起），说明你对系统架构有良好直觉。
3. 主动要求 review 和引导式教学。 shared_ptr 那次你明确说"不要直接指出来，引导我去思考"，说明你在有意识地训练自己的排错能力，而不是单纯要答案。
4. 愿意在困难点上死磕。 pipefs 的 refcnt 问题你前后至少经历了三四轮迭代才彻底解决，过程中加了时序日志、画流程图，调试方法论在进步。

没做好什么
1. 编译错误的排查能力应该更强。 extern "C" 导致的链接问题你至少遇到了三次（tss_set_kernel_stack、v_open、string.libk.o），每次都来问我。这类问题有明确的诊断套路（nm 查符号、看有没有 mangling），遇过一次之后应该能自己处理。
2. 第一版代码的质量有提升空间。 你贴给我的代码经常有比较初级的问题：缺 return 语句、复制粘贴后忘改变量名（TCP hasher 里 remote_port 写了两次）、伪首部长度不包含 payload 等。这些不是能力问题，是写完之后没有逐行 review 自己代码的习惯问题。
3. 并发/竞态思考不够前置。 pipefs 的问题本质上是"引用计数 vs 布尔值"以及"锁的粒度"，这些在设计阶段就应该想清楚，而不是跑出 bug 后才发现。tcp_connect 的锁使用问题也是类似——先发包再拿锁，这种时序问题应该在写代码时就画出来。
4. 对 freestanding 环境的约束估计不足。 用 unordered_map 做全局变量导致 __cxa_atexit 问题、需要浮点但没启用 FPU——这些说明你在写代码时有时会忘记自己不是在写普通应用程序。

本该做什么

建立一个"写完代码后的自检清单"： 每个函数写完后检查：所有路径都有 return 吗？锁是否配对？引用计数增减是否对称？变量名是否复制粘贴后遗忘修改？
遇到链接/编译错误时先用工具自查： nm、objdump -t、readelf 这些是你的朋友。先看符号再来问。
在设计阶段就画出并发时序图： 特别是涉及多进程共享资源（pipe、socket、fd table）的部分，先画"进程 A 做什么→进程 B 做什么→中断来了会怎样"的时序，再写代码。

不该做什么

不该在没有先 review 自己代码的情况下直接贴给我看。 至少一半的 bug 是你自己花 5 分钟逐行读就能发现的。
不该反复在同一类问题上依赖我。 C/C++ 链接问题、freestanding 环境限制这些"交过一次学费"的知识，应该内化成条件反射。


一句话总结：你的系统设计能力和项目推进力远超平均水平，短板在于代码的"最后一公里"质量——写完后的自检、对并发的前置思考、以及对重复性问题的自主解决。 如果你能在"提交代码前多花 10 分钟自查"这件事上养成习惯，对我的依赖可以砍掉至少一半，而项目质量会提升一大截。
